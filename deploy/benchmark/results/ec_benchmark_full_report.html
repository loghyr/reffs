Erasure Coding for pNFS Flex Files

Benchmark report: codec comparison, geometry scaling, degraded reads, SIMD acceleration, and chunk metadata persistence

Executive Summary

This report presents benchmarks for a pNFS Flex Files erasure-coded layout comparing five I/O strategies: plain mirroring (single-DS baseline), Reed-Solomon (RS), Mojette systematic, and Mojette non-systematic erasure coding at 4+2 and 8+2 geometries. Tests span five file sizes (4 KB–1 MB) on four platforms covering two ISAs (aarch64 NEON and x86_64 AVX2), with both SIMD-accelerated and forced-scalar runs plus degraded-1 reads to measure reconstruction overhead. Each result is the mean of five runs.

Central findings: (1) EC write overhead is modest at small-to-mid sizes; (2) degraded reads carry negligible overhead for systematic codecs; (3) at 8+2, Mojette systematic reconstruction stays near-zero while RS grows substantially; (4) overhead ratios are consistent across four platforms and two ISAs; (5) SIMD vs scalar differences are within noise at the current 4 KB shard size (I/O dominates); and (6) chunk metadata persistence (CRC32, owner, lock flags) is now crash-safe via write-temp/fdatasync/rename, ready for the v2 CHUNK benchmark.

5. SIMD Acceleration

The Mojette forward transform includes SIMD fast paths for |p|=1 directions: AArch64 NEON (128-bit, 2×uint64), x86_64 SSE2 (128-bit, baseline fallback), and x86_64 AVX2 (256-bit, 4×uint64). The --force-scalar flag bypasses SIMD dispatch for benchmarking. Seven unit tests verify bit-identical results across all paths.

5.1 SIMD vs scalar: Mojette-sys 4+2 at 1 MB

Figure 7 — SIMD vs scalar, Mojette-sys 4+2, write and read at 1 MB.

At the current 4 KB shard size, SIMD vs scalar differences are within run-to-run variance (±5–20%). The forward transform touches only parity computation; at small shards the grid is small and I/O dominates. The SIMD benefit will become measurable when shard sizes increase to 64 KB+ (when the io_uring large-message constraint is lifted), where encoding math is the bottleneck.

Key result: SIMD correctness is verified end-to-end — NEON on Apple Silicon, AVX2 on Intel and AMD, forced-scalar as reference. All 2,600 verification checks pass across four platforms. The performance optimization is in place and ready for larger workloads.

6. Chunk Metadata Persistence

The Flex Files v2 layout uses CHUNK_WRITE to send both CRC32 checksums and encoded data to each data server. The CHUNK operation lifecycle is:

  1. CHUNK_WRITE — store data via pwrite, record per-block metadata (PENDING)
  2. CHUNK_FINALIZE — transition PENDING → FINALIZED (CRC-validated)
  3. CHUNK_COMMIT — transition FINALIZED → COMMITTED (durable)

6.1 The persistence gap

The initial implementation stored chunk metadata (CRC32, chunk_owner4, block state) in an in-memory array only. Data was written to disk via pwrite, but the metadata that says “this block is COMMITTED with CRC X owned by client Y” was lost on DS restart. This broke the durability guarantee of CHUNK_COMMIT.

6.2 On-disk format

Each inode's chunk metadata is now persisted to <state_dir>/chunks/<inode_ino>.meta using the standard write-temp/fdatasync/rename pattern. The file format is:

FieldSizeDescription
Header32 bytesMagic (CKST), version, nblocks, inode_ino
Block entries32 bytes eachstate, flags, gen_id, client_id, owner_id, payload_id, crc32, chunk_size

Per-block fields:

  • chunk_owner4: gen_id + client_id + owner_id — identifies which client wrote each block
  • crc32: CRC32 checksum as received in CHUNK_WRITE, validated server-side
  • flags: uint32_t with CHUNK_BLOCK_LOCKED = 0x1 (wire format remains chrr_locked as bool<>)
  • chunk_size: actual payload size per block — critical for Mojette, where parity projections have different sizes per direction (B = |p|(Q−1) + |q|(P−1) + 1)

6.3 Persistence triggers

Metadata is written to disk on FINALIZE and COMMIT transitions — the two points where the client has a durability expectation. PENDING state is transient (client can retry the write). On first chunk_store_get() for an inode, the server attempts to load the metadata file; if none exists, a fresh in-memory store is created.

6.4 RocksDB readiness

The fixed-size block records indexed by offset map directly to a key-value store: key = inode_ino:block_offset, value = 32-byte block record. When RocksDB replaces the POSIX backend, the persistence layer changes from file I/O to rocksdb_put() with no structural changes to the chunk_store API.

6.5 Next step: v1 vs v2 benchmark

With metadata persistence in place, the next benchmark compares the v1 (NFSv3 WRITE) and v2 (CHUNK_WRITE) data paths. The v2 path adds: CRC32 computation per chunk on the client, CRC32 validation on the server, the FINALIZE and COMMIT round-trips, and the metadata persistence I/O. The Docker benchmark infrastructure already runs separate DS containers, so the session multiplexing blocker (single-host combined mode) does not apply. This will answer the central question: what is the cost of writing CRC + data to separate stores vs a plain single-blob write?

7. Conclusions

EC overhead is affordable. At 4–64 KB, all EC codecs add 14–37% write overhead. At 1 MB, RS and Mojette-sys reach +53–60%.

Reconstruction is essentially free for systematic codecs. Mojette-sys at 8+2 adds +2% to degraded reads. RS reconstruction grows with k but at I/O-dominated shard sizes the effect is small.

Mojette-sys 8+2 is the recommended operating point. Best read latency (82.6 ms at 1 MB), lowest storage overhead (25%), near-zero reconstruction cost (+2%). RS 4+2 remains the conservative fallback.

Mojette non-systematic is unsuitable for interactive workloads. 4x read overhead at 4+2, 6x at 8+2, due to mandatory inverse transform on every read.

Results are ISA-independent. Four platforms (M4 NEON, i9 AVX2, N100 AVX2, Ryzen 7 AVX2) produce consistent overhead ratios despite 2–6x absolute latency differences.

SIMD is correct but I/O-bound at 4 KB shards. NEON and AVX2 fast paths are verified across all platforms (2,600 operations, zero failures). The encoding benefit will appear at larger shard sizes.

Chunk metadata is now crash-safe. CRC32, chunk_owner4, and lock flags persist via write-temp/fdatasync/rename. The v2 CHUNK benchmark is unblocked.
