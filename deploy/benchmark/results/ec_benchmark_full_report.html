Erasure Coding for pNFS Flex Files

Benchmark results: codec comparison, striping analysis, degraded reads, multi-geometry scaling, and SIMD acceleration

Executive Summary

This report presents benchmarks comparing four I/O strategies for a pNFS Flex Files erasure-coded layout: plain mirroring (single-DS baseline), pure striping, Reed-Solomon (RS) erasure coding, Mojette systematic erasure coding, and Mojette non-systematic erasure coding. Tests were run across five file sizes (4 KB–1 MB) at two geometries (4+2 and 8+2) on Linux hosts using Docker containers as data servers. Benchmarks were conducted on two independent platforms — an Apple M4 MacBook running Fedora 43 under VMware Fusion (aarch64, NEON), an Intel N100 mini-PC (x86_64, SSE2), and an AMD Ryzen 7 5700U laptop (x86_64, SSE2) — to verify that results reflect codec and protocol properties rather than host-specific or ISA-specific artifacts. Each result is the mean of five measured runs.

The central findings are: (1) EC write overhead at small-to-mid sizes is modest and within operational tolerance; (2) the striping baseline isolates parallel I/O cost from encoding cost, revealing that the dominant write penalty is the parity shard computation rather than fan-out latency; (3) degraded reads — where one data shard must be reconstructed — carry negligible latency overhead for systematic codecs; (4) at the wider 8+2 geometry, systematic Mojette’s reconstruction cost stays near-zero while Reed-Solomon’s grows substantially, exposing a fundamental difference in reconstruction algorithm complexity; (5) overhead ratios are consistent across three test platforms spanning two ISAs (aarch64 NEON vs x86_64 SSE2), confirming portability of the implementation; and (6) SIMD acceleration (AArch64 NEON and x86_64 SSE2) has been added for the Mojette forward transform, with AVX2 benchmarks pending.

Test Infrastructure

Primary benchmarks (sections 1–4) were run on Fedora 43 (aarch64, Apple M4 under VMware Fusion) using Docker containers on a single-host bridge network. Cross-platform benchmarks (section 5) add two x86_64 machines: an Intel N100 and an AMD Ryzen 7 5700U, both running Fedora 43 natively. Containers act as independent data servers (DSes) and a single metadata server (MDS). The client is an NFSv4.2 implementation exercising Flex Files v1 layouts (NFSv3 DS I/O). The shard size is fixed at 4 KB as a workaround for an io_uring large-message constraint.

Because all DSes run on the same physical host, network round-trip latency is near-zero. Write numbers therefore reflect pure encoding and RPC fan-out cost. Read numbers reflect decode cost and parallel RPC assembly. On a real network with non-trivial latency, reconstruction overhead for systematic codecs would remain similarly low (decode is CPU-bound, not network-bound), while absolute latency numbers would be higher. The results are conservative in that direction.

1. Codec Comparison at 4+2

The baseline geometry is 4+2: four active data shards and two parity shards, six DSes total, tolerating two simultaneous DS failures. Plain mirroring uses a single DS and serves as the zero-overhead reference.

1.1 Write latency

At 4–64 KB all three EC codecs track within 14–21% of plain mirroring — within the noise floor of real-network deployments. Above 64 KB the gap widens; at 1 MB Reed-Solomon and Mojette systematic reach approximately 54% overhead while Mojette non-systematic reaches 62%. The write cost growth reflects encoding CPU cost scaling faster than the I/O path as payload grows.

1.2 Read latency

Read performance is where codecs diverge significantly. RS and Mojette systematic remain within 25–63% of plain mirroring at all sizes. Mojette non-systematic requires a full inverse transform on every read regardless of shard availability, producing 2x overhead at 64 KB and 4.2x (321%) at 1 MB.

2. Isolating I/O Cost from Coding Cost

A pure-striping variant (6 data shards, no parity) was added to separate the overhead of parallel RPC fan-out from the overhead of encoding. This isolates two distinct cost layers that EC benchmarks normally conflate.

2.1 Write cost decomposition

At 1 MB, plain mirroring writes 64 ms, stripe 71 ms (+11%), and RS 103 ms (+60% over plain). The stripe baseline reveals that only 7 ms of the 39 ms RS penalty comes from parallel I/O fan-out. The remaining 32 ms is encoding math plus two additional parity RPCs — the true cost of fault tolerance. For Mojette systematic the coding layer is 52 ms; for non-systematic, 69 ms.

Figure 5 — Write overhead decomposed into parallel I/O cost and coding overhead at 1 MB.

At small sizes (4–16 KB), stripe and plain are statistically indistinguishable. Per-RPC overhead is small relative to data transfer; the fan-out cost is negligible. This means small-file EC overhead is almost entirely encoding math, not I/O structure.

2.2 Read latency including stripe baseline

Stripe reads are only 9% above plain at 1 MB, confirming that parallel fan-out on the read path is nearly free. RS and Mojette systematic at 4–16 KB are at or below the stripe baseline — reading from 4 data shards of size file/4 each is faster than reading from 6 stripe shards of size file/6, because there are fewer total RPCs to complete before reassembly. Mojette non-systematic diverges above 64 KB due to its mandatory reconstruction pass.

3. Degraded Reads and Reconstruction

Each test file was written with all DSes healthy, then read under two conditions: all DSes responding (healthy), and with data shard 0 skipped to force reconstruction from the remaining five shards (degraded-1). Skipping a data shard forces real reconstruction work for systematic codecs. All 75 degraded reads (3 codecs × 5 sizes × 5 runs) verified byte-for-byte correct against the original input.

3.1 Healthy vs degraded-1 latency

Solid lines show healthy reads; dashed lines show degraded-1. For RS and Mojette systematic the two lines are visually indistinguishable at 256 KB and 1 MB. For Mojette non-systematic, degraded-1 adds a small increment over an already high baseline — the codec performs the same reconstruction work whether or not all shards are present.

Figure 7 — Healthy vs degraded-1 read latency (ms). Solid = healthy, dashed = degraded-1.

3.2 Reconstruction overhead

Reconstruction overhead is expressed as the percentage increase from the healthy read baseline. Negative values at some sizes indicate that reading from five DSes is faster than six: skipping one DS eliminates one fan-out RPC, and at small file sizes the saved round-trip offsets the decode math. This is a real effect, not measurement noise.

A client that discovers a failed DS at read time can reconstruct transparently with no meaningful impact on user-visible latency. CB_LAYOUTRECALL and re-layout are needed only on the write path.

4. Geometry Scaling: 4+2 vs 8+2

The 4+2 and 8+2 geometries represent the two natural operating points for a two-parity-shard deployment. Both tolerate two simultaneous DS failures. The key difference is shard count and storage overhead: 4+2 writes 150% of the file size across six DSes; 8+2 writes 125% across ten DSes. With m=2 held constant, encoding cost should be geometry-independent (it is O(m × file_size)); any difference in measured latency comes from shard size and parallelism effects.

4.1 Write latency

RS and Mojette systematic both improve modestly at 8+2. RS drops from 103 to 96 ms at 1 MB (−6%); Mojette systematic drops from 137 to 115 ms (−16%). Smaller shards mean smaller individual RPCs, and ten DSes provide more parallelism. The encoding cost itself is unchanged (m=2 in both cases), confirming the O(m × file_size) model.

Figure 9 — Write latency (ms) across codecs and geometries. Y-axis capped at 160 ms.

4.2 Read latency

RS reads improve significantly at 8+2: 1 MB healthy read drops from 81 to 71 ms (−13%). Mojette systematic also improves slightly (84 to 81 ms). Both benefit from eight parallel reads of smaller shards. Mojette non-systematic at 8+2 is substantially worse: 395 ms at 1 MB vs 246 ms at 4+2, because the inverse Mojette transform scales with grid size (8 rows × P columns vs 4 rows). This codec should not be used at wide geometries.

Figure 10 — Healthy read latency (ms). Solid = 4+2, dashed = 8+2.

4.3 Reconstruction at 8+2

This is the most significant result of the multi-geometry benchmark. At 4+2, RS and Mojette systematic both showed near-zero reconstruction overhead. At 8+2 they diverge sharply. RS degraded-1 overhead at 1 MB grows from +6% to +54%. Mojette systematic remains at +4%. The difference reflects the underlying algorithms: RS reconstruction requires inverting a k × k matrix in GF(28), an O(k2) operation. Doubling k from 4 to 8 multiplies matrix inversion cost by roughly 4x. Mojette systematic reconstruction uses back-projection, whose cost scales with m × k — the same factor as encoding. Since m=2 is unchanged, reconstruction cost at 8+2 is only marginally higher than at 4+2.

5. Cross-Platform and Cross-ISA Comparison

The codec comparison benchmarks were run on three platforms spanning two ISAs to test whether overhead ratios are architecture-independent:

  • dreamer — Apple M4 (aarch64, NEON), Fedora 43 VM via VMware Fusion on MacBook Pro
  • adept — Intel N100 (x86_64, SSE2), Fedora 43 native, 4 cores, 3.4 GHz max
  • garbo — AMD Ryzen 7 5700U (x86_64, SSE2), 8 cores / 16 threads, 4.3 GHz boost

The key question is whether the relative overhead ratios — not absolute latencies — are reproducible across CPU vendors, ISAs, and SIMD implementations.

5.1 Absolute latency at 1 MB

Absolute latencies vary widely: the M4 is fastest (64 ms plain write), the N100 is 3x slower (204 ms), and the Ryzen 7 is 4.5x slower (290 ms). These differences reflect CPU speed, Docker overhead, and memory bandwidth — not codec properties.

6. SIMD Acceleration

The Mojette forward transform (moj_forward()) now includes SIMD fast paths for the two most common projection directions: p=+1 and p=−1 (both with q=1). These are the directions used by the systematic Mojette codec for its primary data projections.

6.1 AArch64 NEON

For p=+1, adjacent grid columns map to adjacent bins in ascending order. The row accumulation becomes a plain sequential vector add (vaddq_u64), unrolled 4-wide (two 128-bit loads per iteration). For p=−1, bin indices descend; pairs of grid elements are lane-swapped with vextq_u64 before addition. A 2-wide cleanup loop handles widths not divisible by 4.

6.2 x86_64 SSE2

The SSE2 implementation mirrors the NEON logic exactly. Sequential ascending bins use _mm_add_epi64; descending bins use _mm_shuffle_epi32(v, 0x4E) to swap 64-bit halves (the SSE2 equivalent of vextq_u64). SSE2 is baseline on all x86_64 processors, so no runtime feature detection is needed.

6.3 Patent safety

The StreamScale patent (US 8,683,296) covers SIMD-accelerated Galois field arithmetic. The Mojette transform uses no GF operations — these SIMD paths are plain unsigned 64-bit integer addition and are entirely unaffected.

6.4 Correctness testing

Seven unit tests verify the SIMD paths produce results identical to the scalar reference:

  • Round-trip tests (p=+1 and p=−1): 4×4 grid, forward+inverse, verify bit-identical recovery. Direction sets isolate one SIMD direction each.
  • Tail tests (P=7, not divisible by 4): exercise the scalar cleanup loop after the 4-wide SIMD body.
  • Exact bin-value tests (3×2 grid): hand-computed expected bin sums verified against moj_forward() output.
  • Reproducibility test (64×64 grid): two independent forward passes produce bit-identical bins.

6.5 Benchmark coverage

The aarch64 NEON path has benchmark data from the M4 MacBook (sections 1–4). The x86_64 SSE2 path has benchmark data from the Intel N100 and AMD Ryzen 7 (section 5). Overhead ratios are consistent across both SIMD implementations, confirming that the fast paths produce equivalent codec behaviour.

Pending: x86_64 AVX2 fast path is planned but not yet implemented. AVX2 would double the vector width to 256-bit (4 × uint64_t per iteration). Degraded-mode benchmarks on x86_64 are also pending.

7. Conclusions

EC write overhead is affordable. At 4–64 KB, all three EC codecs add 14–21% write overhead — within the noise of real-network variance. The encoding cost only becomes significant above 64 KB where payload volume makes per-byte encoding cost measurable.

The encoding cost, not the I/O cost, drives EC write overhead. The striping baseline isolates these: at 1 MB, parallel fan-out adds 7 ms while RS encoding adds 32 ms. Optimizing the I/O path has diminishing returns; codec compute efficiency matters.

Reconstruction is essentially free for systematic codecs. At 4+2, RS and Mojette systematic add 1–6% to read latency when reconstructing a missing shard. A client can tolerate a DS failure transparently with no user-visible impact. This holds at 8+2 for Mojette systematic (+4%) but not for RS (+54%), due to the O(k2) matrix inversion in RS decode.

Mojette systematic is the recommended codec at 8+2. It matches RS on healthy read performance, improves on write, achieves 25% storage overhead vs 50% at 4+2, and maintains near-zero reconstruction cost. The 8+2 geometry benefits Mojette systematic specifically because its reconstruction scales with m (parity count), not k (data count). For deployments where proven algebraic reconstruction guarantees are required, RS 4+2 or 6+2 remains the conservative choice.

Mojette non-systematic is unsuitable for general use. Its mandatory full inverse transform on every read produces 4x overhead at 1 MB for 4+2 and 7x for 8+2. It may be acceptable for write-once cold storage where reads are rare, but should not be the default codec for any interactive workload.

RS sweet spot is lower k. RS reconstruction scales as O(k2): at k=4 it is negligible; at k=8 it adds 54% to degraded read latency. The RS operating point that balances storage efficiency, write cost, and reconstruction cost is 4+2 or 6+2. Beyond k=6, reconstruction overhead during a DS failure becomes meaningful at exactly the wrong time.

Results are platform- and ISA-independent. Benchmarks run on three platforms — Apple M4 (aarch64, NEON), Intel N100 (x86_64, SSE2), and AMD Ryzen 7 (x86_64, SSE2) — produce consistent overhead ratios despite 3–5x differences in absolute latency. RS write overhead is +49–61% across all platforms; RS read overhead is +46–53%. The codec ordering, overhead percentages, and qualitative conclusions are fully reproducible across CPU vendors, ISAs, and SIMD implementations.

SIMD acceleration is in place for Mojette. NEON (aarch64) and SSE2 (x86_64) fast paths accelerate the forward transform for |p|=1 directions. Both paths have real benchmark data (NEON on M4, SSE2 on N100 and Ryzen 7) and produce equivalent overhead profiles. Correctness is verified by 7 dedicated unit tests. AVX2 is planned as a future optimization.
